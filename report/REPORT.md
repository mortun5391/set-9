# Отчёт по исследованию сортировок строк

Автор: Халилбеков Х. А.

## Эксперимент

Исследованы шесть алгоритмов: обычные Quicksort и Mergesort, тернарный String Quicksort, String Mergesort с LCP, MSD Radix Sort и гибрид MSD Radix Sort с переключением на String Quicksort для подмассива размером менее 74.

Использованы четыре набора строк: случайный, обратно отсортированный, почти отсортированный и дополнительный стресс-тест с общим префиксом длины 70. Размер массива изменяется от 100 до 3000 с шагом 100. Время усредняется по 11 запускам. Результат каждого запуска сверяется со стандартной сортировкой C++.

## Итоги при n = 3000

| Набор | Лучший алгоритм | Среднее время, мкс | Ускорение относительно лучшего стандартного |
|---|---|---:|---:|
| random | MSD Radix Sort | 242.1 | 3.01x |
| reverse | MSD + String Quicksort | 213.4 | 1.60x |
| nearly_sorted | MSD + String Quicksort | 191.2 | 1.68x |
| common_prefix | String Quicksort | 850.1 | 7.57x |

## Сопоставление с теорией

Обычные Quicksort и Mergesort имеют сравнительную нижнюю границу Ω(n log n), но для строк каждое сравнение дополнительно зависит от длины общего префикса. Поэтому на наборе `common_prefix` число посимвольных сравнений у Mergesort при n = 3000 равно 2 209 935, а у Quicksort - 2 973 085.

Тернарный String Quicksort работает по символам и группирует строки с одинаковым текущим символом, поэтому на длинных общих префиксах не повторяет полное сравнение строк на каждом уровне сравнивающей сортировки. На `common_prefix` он выполняет 243 625 посимвольных сравнений и становится быстрее лучшего стандартного алгоритма в 7.57 раза.

String Mergesort с LCP сохраняет длины уже известных общих префиксов и начинает следующее сравнение с известной позиции. Это уменьшает число сравнений: на `common_prefix` 237 736 посимвольных сравнений, то есть около 10.8% от обычного Mergesort. По времени он всё равно уступает String Quicksort из-за дополнительных массивов LCP и операций слияния.

MSD Radix Sort близок к оценке O(W + Rn) для просмотренных символов, где W - суммарное число фактически обработанных символов, R = 74 - мощность алфавита. В таблице для него `char_comparisons = 0`, потому что алгоритм не сравнивает символы попарно, а распределяет строки по корзинам; для интерпретации добавлена метрика `char_inspections`. Гибрид с переключением на String Quicksort уменьшает накладные расходы на малых подмассивах и поэтому выигрывает на обратно и почти отсортированных данных.

## Выводы

Стандартные сортировки заметно проигрывают на строках с длинными совпадающими префиксами, поскольку повторно просматривают одинаковые начальные части строк. Тернарный String Quicksort показал лучший результат на префиксном наборе. MSD Radix Sort стал лучшим на случайном наборе, а гибрид MSD со String Quicksort - на обратно и почти отсортированных данных. String Mergesort с LCP особенно сильно уменьшает число посимвольных сравнений: на наборе `common_prefix` оно составляет около 10.8% от обычного Mergesort.

Подробные численные результаты для размера 3000 расположены в `data/summary_at_3000.csv`. Полный файл результатов расположен в `data/results.csv`, сжатая копия - в `data/raw/`. Графики сохранены в `figures/`.
